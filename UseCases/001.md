# Use Case: Extract and Markup Book Content

**Context of Use:** A public domain book has been physically digitized. The E-book Producer needs to convert these static images into semantically structured text and isolated illustration files without losing visual formatting context, managing complex multi-pass interfaces, or worrying about data loss.

**Scope:** The native macOS Digitization App.

**Level:** User-Goal

**Primary Actor:** The E-book Producer

**Precondition:** A sequential folder of digitized page images (JPG, PNG, or TIFF) exists on the local macOS file system.

**Minimal Guarantees:** 1. The original raw image files remain completely unaltered on the disk.
2. The user's work state is preserved up to the last millisecond via continuous background autosave.

**Success Guarantees:** 1. The book's primary text flow is extracted, structurally tagged, and exported as a well-formed XHTML document.
2. Repeating page artifacts (page numbers, running headers) are identified and excluded from the continuous XHTML text flow.
3. Illustrations are cropped, saved to disk as high-resolution assets, and accurately referenced via `<img>` tags within the XHTML sequence.
4. All text, structural blocks, and image placements are explicitly validated or modified by the Producer via a single, synchronized review workspace before export.

### Main Success Scenario

1. **The Producer points the system** to a folder of sequential scanned images.
2. **The System initializes a new project**, establishing a continuous background autosave state on disk.
3. **The System processes the images asynchronously**, utilizing the Vision framework to extract text and analyze layout geometry.
4. **The System categorizes layout blocks** into three distinct semantic zones:
* *Body Content:* The primary continuous flow of the book (encompassing paragraphs, headings, blockquotes, and inline formatting).
* *Marginalia:* Isolated notes or footnotes outside the main text flow.
* *Page Artifacts:* Repeating running headers, footers, and page numbers.


5. **The System isolates non-text illustrations**, generating bounding boxes over them and mapping their exact sequential position relative to the surrounding Body Content.
6. **The System presents a synchronized, dual-pane review interface** to the Producer:
* *The Image Stream:* A continuous, vertically scrolling column of the raw source images. Images are sized to a configurable percentage of the window height (e.g., 50-80%) to ensure boundaries between adjacent pages are always visible. Any detected illustrations display a native-feeling overlay rectangle.
* *The Plaintext Editor:* A continuous, vertically scrolling plaintext editor containing the full XHTML document. The text flow maps line-for-line with the physical layout of the book, utilizing carriage returns within logical container tags (e.g., a single `<p>` tag wrapping across multiple lines and page boundaries).
* *Synchronized Behavior:* Scrolling either pane automatically scrolls the other, keeping the visual image context perfectly aligned with the text block being reviewed.


7. **The Producer reviews the raw XHTML** against the visual Image Stream, verifying layout, transcription accuracy, and illustration boundaries.
8. **The Producer confirms completion** and triggers the Export command.
9. **The System writes the finalized, clean XHTML file** and the asset folder to the local disk.

### Extensions

**3a. The System encounters a severely degraded page or fails to extract text:**

1. The System automatically executes a high-accuracy fallback OCR pass.
2. If the fallback fails, the System flags the page as "Manual Entry Required" and creates an empty text block within the project file.
3. The System resumes processing subsequent pages.

**6a. The Project folder containing the raw source scans is moved or renamed on disk:**

1. The System detects the broken path upon project launch, flags the missing assets, and disables editing.
2. The System prompts the Producer with a native macOS folder picker to locate the new directory.
3. The Producer selects the folder; the System restores the image stream bindings and enables editing.

**7a. The Producer identifies an incorrect structural tag or text error:**

1. The Producer edits the text directly in the Plaintext Editor.
2. *Shortcut Magic:* The Producer can highlight a block of text and use native macOS shortcuts (e.g., `Cmd+B` for bold, `Cmd+I` for italics) to wrap the text selection in XHTML tags (e.g., `<strong>`, `<em>`).
3. The System autosaves the text manipulation instantly.

**7b. The Producer notices the source image in the Canvas is inverted or misaligned:**

1. The Producer triggers a rotation or flip command using a standard shortcut.
2. The System updates the Canvas and persists the new image orientation.
3. *System Decision:* * If the Producer has **not** manually edited the text for this page, the System automatically re-runs the text extraction on the newly oriented image and updates the Plaintext Editor.
* If the Producer **has** manually edited the text, the System preserves the text exactly as is.

**7c. The Producer needs to adjust, create, or delete an illustration boundary:**

1. *To Modify:* The Producer clicks an existing bounding box rectangle on the Image Stream canvas. The box becomes active with resize handles. The Producer drags the corners to adjust the crop. The System updates the cropped file asset on disk.
2. *To Create:* The Producer clicks and drags an empty area of the Canvas to draw a new rectangle. The System crops the asset, determines its sequential position relative to the surrounding text coordinates, and automatically inserts a matching `<img src="..."/>` tag into the Plaintext Editor flow.
3. *To Delete:* The Producer selects an active bounding box and triggers the delete command (via keyboard shortcut, menu item, or toolbar button). The System deletes the cropped file asset from disk and automatically sweeps through the Plaintext Editor to remove the corresponding `<img>` tag.
4. The System updates the persistent system state.

**7d. The Producer identifies a page where the initial extraction failed, but the System did not detect the failure:**

1. The Producer triggers a "Force Re-Extract" command via keyboard shortcut, menu item, or toolbar button.
2. The System automatically executes a high-accuracy fallback OCR pass on the current page.
3. The System overwrites the Plaintext Editor for that specific page with the new results.
4. The Producer resumes the standard visual review.
